Короткий ответ: **минимум 3 ноды (брокера)**.

Это стандартная рекомендация Apache Kafka для любого продакшн-кластера. Вот подробное объяснение, почему именно 3, и какие нюансы нужно учитывать.

### 1. Почему минимум 3 ноды? (Логика кворума)
Kafka (как и многие распределенные системы на базе Raft/ZooKeeper) работает по принципу **кворума** (мажоритарности).
*   **1 нода:** Если она упадет — кластер «умрет». Данные потеряны.
*   **2 ноды:** Если упадет одна — кластер потеряет кворум (1 из 2). Произойдет **потеря данных** или остановка кластера, так как невозможно найти большинство.
*   **3 ноды:** Если упадет одна — осталось 2. Это **2/3 (кворум)**. Кластер продолжает работать, данные доступны.
*   **5 нод:** Если упадет одна — осталось 4. Кворум сохраняется. Также позволяет учитывать отказ целой «зоны» (rack), если ноды физически разнесены.

**Итог:** 3 ноды — это математический минимум для отказоустойчивости (HA).

---

### 2. Критически важная настройка: Replication Factor (RF)
Количество нод — это не все. Вам нужно настроить количество реплик для каждого топика.

*   **Replication Factor (RF)** — сколько копий данных хранится на разных нодах.
*   **Минимум:** `replication.factor` должен быть равен количеству доступных брокеров в кластере.
*   **Практика:** Для кластера из 3 нод `RF = 3`. Для кластера из 5 нод `RF = 3` (достаточно, чтобы 2 из 3 реплик были на живых нодах) или `RF = 5` (если важна защита от отказа целой стойки).

> **Важно:** Если `RF` меньше количества вышедших из строя нод, данные потеряются.

---

### 3. Контроллеры (KRaft vs Zookeeper)
С версии 2.8+ Kafka использует встроенный режим хранения метаданных **KRaft** (вместо вынесенного Zookeeper).

*   **В KRaft:** Контроллеры выбираются из самих брокеров. Вам нужно минимум 3 брокера, чтобы выбрать контроллер, который выдержит отказ.
*   **В старом Zookeeper:** Вам нужно было 3 брокера Kafka + 3 ноды Zookeeper. К этому стоит относиться аккуратно, так как ZK потребляет много ресурсов.

---

### 4. Когда нужно больше 3 нод? (5, 7, 9...)
Хотя 3 ноды делают кластер «надежным», для крупных инфраструктур часто используют 5 нод.
**Плюсы 5 нод:**
1.  **Rack Awareness (Авансирование стойки):** Вы можете разместить 3 ноды в одной стойке, а 2 в другой. Если одна стойка сгорит, данные потеряются (если RF=3). Если RF=3 и ноды в разных стойках, вы переживете отказ стойки.
2.  **Баланс нагрузки:** Больше нод = больше дисков = больше пропускной способности (IOPS) и места.
3.  **Масштабируемость:** Легче добавить ноды в будущем без перерасчета кворума.

---

### 5. Чего НЕ делать (Типичные ошибки)
1.  **Кластер из 1 ноды:** Это не кластер, это одна точка отказа. Можно использовать только для тестов в `localhost`. В продакшене — категорически нет.
2.  **Кластер из 2 нод:** Не рекомендуется. Риск потери данных при падении любого узла.
3.  **Меньше 3 реплик (RF=2) на кластере из 3 нод:** Технически возможно, но при падении одной ноды, где лежит одна из реплик, вы потеряете часть данных (если ISR сужается).
4.  **Размещение всех нод на одной машине:** Если вы ставите 3 брокера на 1 сервер с 6 ядрами, и сервер упадет — всё. Надежность нужно строить на *физических* отказах оборудования.

### Итоговая рекомендация

| Сценарий | Минимум нод | Рекомендация | Replication Factor |
| :--- | :--- | :--- | :--- |
| **Dev / Тесты** | 1 | Можно 1 (но данные не защищены) | RF=1 |
| **Продакшен (Минимум)** | **3** | **3 ноды в разных физ. стойках (если возможно)** | **RF = 3** |
| **Продакшен (Оптимально)** | **5** | **5 нод (3 в одной стойке, 2 в другой)** | **RF = 3** (или 5) |
| **Масштабный кластер** | 7+ | 7+ нод для балансировки и отказоустойчивости | RF = 3 |

**Резюме:** Чтобы сделать **надежный** кластер, ставьте **3 брокера**. Обязательно настройте `replication.factor` минимум в 3. И лучше всего, если эти 3 брокера будут стоять на разных физических серверах (или в разных дата-центрах).

### 1. Как распределяются IP-адреса в Kafka

В Kafka (и в Docker) важно различать два типа IP-адресов:

1.  **Internal/Bind IP (Локальный IP):** IP, на который Kafka "слушает" входящие соединения внутри контейнера/сервера. Обычно это `0.0.0.0` (все интерфейсы) или конкретный внутренний IP.
2.  **Advertised IP (Объявленный IP):** IP, который Kafka сообщает клиентам и другим брокерам, чтобы они могли подключиться.
    *   **Внутри кластера:** Клиенты используют `advertised.listeners` (например, IP самого сервера или имя хоста в Docker-сети).
    *   **Для внешнего доступа:** Обычно мы используем публичный IP или IP Docker-хоста (`host.docker.internal` для тестов), чтобы клиенты находили брокер по одному адресу.

**Важно:** В кластере из 3 нод каждый брокер должен знать IP-адреса всех остальных нод для формирования кворума (KRaft).

### 4. Важные нюансы для надежности

1.  **`controller.quorum.voters` — самая хитрая настройка.**
    В KRaft этот параметр должен быть **идентичным** на всех узлах. Он описывает, какие узлы будут участвовать в контроллерах. Если вы добавите 4-ю ноду, вам придется обновить эту строку конфигурации на всех 3 существующих нодах.
    *   *Совет:* Если это сложно, можно использовать Zookeeper, но это менее актуально для новых версий Kafka.

2.  **`advertised.listeners` vs `listeners`.**
    *   `listeners`: `0.0.0.0:9092` (внутри контейнера слушает всё).
    *   `advertised.listeners`: `IP_СЕРВЕРА_1:9092` (клиенту сообщает, куда подключиться).
    *   Если вы не укажете `advertised.listeners`, Kafka использует IP самого контейнера (изнутри сети), и внешние клиенты не смогут подключиться.

3.  **Балансировка нагрузки (Load Balancer).**
    Клиентам лучше подключаться не к IP каждой ноды, а к Load Balancer (NGINX, HAProxy), который распределяет запросы на 3 IP. Но если подключаться напрямую, используйте `bootstrap.servers=192.168.1.10:9092,192.168.1.11:9092,192.168.1.12:9092`.

4.  **Разные физические диски.**
    В `log.dirs` лучше указывать разные диски на разных серверах, но в Docker это часто `kafka-data` volume. Убедитесь, что на каждом сервере есть достаточно места.

Использование **Zookeeper** как отдельного сервиса для управления метаданными кластера — это классический подход (до появления KRaft в Kafka 2.8+).

**Важно:** Zookeeper требует своих собственных 3 узлов для обеспечения отказоустойчивости. Таким образом, для надежного кластера Kafka на базе Zookeeper вам понадобятся минимум **3 узла Zookeeper + 3 узла Kafka**.
Если у вас 3 физических сервера, на каждом из них нужно запустить по одному контейнеру Zookeeper и по одному контейнеру Kafka.

Ниже приведена конфигурация для 3 физических серверов.

### Архитектура
Каждый физический сервер будет содержать два Docker-контейнера:
1.  **Zookeeper** (порт 2181)
2.  **Kafka** (порт 9092)

---

### 1. Файл `docker-compose.yml`
Этот файл запускается на **каждом** из 3 серверов. Вам нужно будет только подставить свои IP-адреса и имена хостов.

```yaml
version: '3.8'

services:
  zookeeper:
    image: zookeeper:3.8.1
    container_name: ${ZK_NAME}
    hostname: ${ZK_HOSTNAME}
    environment:
      ZK_SERVER_ID: ${ZK_ID}
      ZK_HOST_IP: ${ZK_HOST_IP}
    ports:
      - "${ZK_HOST_IP}:2181:2181" # Биндинг по внешнему IP
    volumes:
      - zookeeper-data:/data
    networks:
      - kafka-net

  kafka:
    image: apache/kafka:3.7.0
    container_name: ${KAFKA_NAME}
    hostname: ${KAFKA_HOSTNAME}
    environment:
      KAFKA_NODE_NAME: ${KAFKA_NODE_NAME}
      KAFKA_BROKER_ID: ${KAFKA_BROKER_ID}
      KAFKA_ADVERTISED_LISTENERS: ${KAFKA_ADVERTISED_LISTENERS}
      KAFKA_LISTENER_SECURITY_PROTOCOL_MAP: ${KAFKA_LISTENER_SECURITY_PROTOCOL_MAP}
      KAFKA_PROCESS_ROLES: broker
      KAFKA_CONTROLLER_QUORUM_VOTERS: # Для ZK режим: здесь пусто (нет контроллеров)
      KAFKA_OFFSETS_TOPIC_REPLICATION_FACTOR: 3
      KAFKA_MIN_INSYNC_REPLICAS: 2
      KAFKA_ZOOKEEPER_CONNECT: ${ZOOKEEPER_CONNECT} # Ключевой параметр для ZK
      KAFKA_LOG_DIRS: /var/lib/kafka/data
      KAFKA_NUM_PARTITIONS: 3
    ports:
      - "9092:9092"
    volumes:
      - kafka-data:/var/lib/kafka/data
    networks:
      - kafka-net

volumes:
  zookeeper-data:
  kafka-data:

networks:
  kafka-net:
    driver: bridge
```

---

### 2. Файл `zookeeper.properties`
Этот файл нужно скопировать в контейнер `zookeeper`. Он хранит настройки репликации между узлами Zookeeper.

```properties
# Базовые настройки
tickTime=2000
initLimit=10
syncLimit=5
dataDir=/data
clientPort=2181
admin.enableServer=false
# Включить статические настройки
4lw.commands.whitelist=*

# Конфигурация кворума (физические IP и порты)
server.1=${ZK_HOST_IP}:2888:3888
server.2=${ZK_HOST_IP}:2888:3888
server.3=${ZK_HOST_IP}:2888:3888

# Файл myid должен лежать в директории /data
myid=${ZK_SERVER_ID}
```
> **Внимание:** В строках `server.1`, `server.2`, `server.3` указывайте **реальные IP-адреса** серверов (или доменные имена, если есть DNS), а не `localhost`.

**Файл `myid`** (для Zookeeper)
В каждом контейнере Zookeeper в директории `/data` должен лежать файл `myid` с одним числом (ID узла), соответствующим `server.x`.
Это файл создается вручную перед запуском.

*Пример для сервера 1:*
```bash
# На сервере 1
docker exec -it zookeeper-1 sh -c "echo '1' > /data/myid"
```

### 3. Файл `kafka.properties`
Если вы решите не передавать настройки через `environment` в `docker-compose`, используйте этот файл. В `docker-compose.yml` можно добавить `command: /opt/kafka/bin/kafka-server-start.sh config/server.properties` и указать путь к этому файлу.

```properties
broker.id=${KAFKA_BROKER_ID}
# Кворум Zookeeper
zookeeper.connect=${ZOOKEEPER_CONNECT}
zookeeper.connection.timeout.ms=6000
zookeeper.session.timeout.ms=6000
zookeeper.sync.timeout.ms=3000
group.initial.rebalance.delay.ms=0
# Настройки сетки
listeners=PLAINTEXT://0.0.0.0:9092
advertised.listeners=PLAINTEXT://${KAFKA_ADVERTISED_LISTENERS}
listener.security.protocol.map=PLAINTEXT:PLAINTEXT
inter.broker.protocol.version=3.7
# Данные
log.dirs=/var/lib/kafka/data
default.replication.factor=3
min.insync.replicas=2
auto.create.topics.enable=true
num.network.threads=3
num.io.threads=8
socket.send.buffer.bytes=102400
socket.receive.buffer.bytes=102400
socket.request.max.bytes=104857600
# Роли
process.roles=broker
```
